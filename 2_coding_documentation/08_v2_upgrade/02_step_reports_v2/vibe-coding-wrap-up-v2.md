# v2「vibe coding」过程收尾（用于导出 PDF）

日期：2026-04-21  
项目：小饭桌（大学生做饭学习规划师）v2 演示版

## 1）vibe coding 的工作方式（本项目实践）
- **增量迭代**：每轮只做一个明确步骤，避免“一次做完 2.0”导致返工与回归。
- **先壳后页**：先把桌面端壳（SideNav + 主内容区）稳定，再逐页对齐高保真。
- **先可演示再完美**：优先保证“可进入、可点击、可闭环”，再做氛围与动效增强。
- **mock-only / RAG-ready**：用本地 mock 贯通链路，同时把 params/prompt/retrieval/provider 抽象出来，为后续真实接入留接口。

## 2）关键取舍（为什么这么做）
- **为什么用 `HashRouter`**：静态部署/课堂演示更稳，避免刷新 404；代价是 URL 带 `#/`。
- **为什么先做结果页**：把“loading/空/错/成功列表/再生成/返回改条件”从首页迁出，首页更聚焦输入与约束。
- **为什么音效/彩带属于体验增强**：不改变业务数据，只增强“确认感/惊喜感”，且可替换素材、可降级（reduced motion / fallback beep）。
- **为什么“最近做过”即使为空也必须完整**：它是学习闭环的“沉淀页”；空状态清晰能引导用户用“我做了”补齐记录。

## 3）常见坑与解决策略（给下一次复用）
- **Windows PowerShell 拦截 `npm`**：优先用 `npm.cmd`；或在非受限环境调整 ExecutionPolicy。
- **音效替换后仍播放旧声音**：浏览器缓存导致，做硬刷新或清缓存；必要时改文件名并同步 `SOUND_ASSETS.success`。
- **响应式异常（900–1120px）**：带侧栏时内容区会变窄，双栏页面必须把断点提到更合理的宽度，避免步骤被压成窄条。
- **lint 噪音**：避免 lint 扫描构建缓存（如 `.vite-cache`），否则误报大量 vendor 代码错误。

## 4）最终交付侧的“可验证性”清单
- `npm.cmd run build` 必须通过
- 首页/结果页/详情页/清单/收藏/最近/贴士/设置可进入
- 关键闭环可演示：
  - 生成 → 结果 → 详情
  - 详情选中 → 加入待购 → 清单详情 CRUD
  - 收藏跨页同步
  - 我做了 → 最近做过

## 5）后续扩展建议（不在本轮做）
- 真正接入 LLM provider：只替换 `src/services/ai/llmProvider.js` 的实现与密钥注入方式
- 真正接入 RAG：替换 `retrievalService` 为向量检索/知识库服务；保留 `promptBuilder` 作为统一拼装层
- 将“结果页状态”持久化（可选）：为课堂演示增加刷新后仍可回看候选菜谱的稳定性
