# Reflection Log（v2）

| Date | Step | Reflection |
|---|---:|---|
| 2026-04-18 | 2.0-01 | 通过 media query 做“桌面壳增强”，能在不推翻移动端结构前提下逐步升级；但必须明确“哪些入口桌面可见/移动端如何到达”，否则会出现不可达问题。 |
| 2026-04-18 | 2.0-01 | 左导航更适合 2.0 的网页信息架构（更多栏目 + 状态徽标）；移动端保留底部导航降低学习成本，双壳共存是风险更低的过渡方案。 |
| 2026-04-18 | 2.0-02 | 首页首屏越像“表单 + CTA”，用户越清楚下一步；把长列表迁出首页能提升首屏效率，但必须配套结果页与清晰入口，否则会造成“生成了但看不到”的困惑。 |
| 2026-04-20 | 2.0-03 | 结果页作为中间层，把“列表展示/再生成/返回改条件”收敛到一处，让首页专注输入与约束；也更符合课堂演示路径（输入→对比候选→详情学习）。 |
| 2026-04-20 | 2.0-03 | 在受限 Windows 环境中，运行可复现性本身就是交付的一部分：需要把 `npm.cmd`、端口冲突处理、build 权限规避方案写进文档，避免“同学照做失败”。 |
| 2026-04-20 | 2.0-04 | “随机弹窗 + 成功提示音”能显著提升演示反馈与操作确定性，但应保持可替换、可关闭、且不影响核心业务链路；因此把音效独立成 service + constants。 |
| 2026-04-20 | 2.0-05 | 详情页把“选中→加入清单→结果确认”串起来后，用户更容易形成可操作的学习闭环；但数量合并与默认全选的 UX 取舍需要在课堂演示中明确说明。 |
| 2026-04-20 | 2.0-07 | 辅助沉淀页（收藏/最近/贴士）不一定有“新功能”，但必须可达、可读、可解释；这是 2.0 演示的完整性要求。 |
| 2026-04-20 | 2.0-09 | 体验增强应优先“可撤销、低风险、无业务耦合”：用 CSS 动效/插画占位/微交互统一手感，比引入动画库更稳。 |
| 2026-04-21 | 2.0-10 | P0 级的视觉与布局修正应优先“收敛变量 + 少量全局替换”，避免到处硬编码；把橙色仅作为主强调能保证克制的产品感，同时让 CTA/当前态更明确。 |
| 2026-04-21 | 2.0-11 | 结构清理类 P0 改动要优先“删冗余入口 + 收束主路径”：移除占位搜索/重复标题/伪功能开关，比继续加功能更能提升 2.0 网页化完成度与演示一致性。 |
| 2026-04-21 | 2.0-12 | “精修”不等于堆效果：先把 spacing、重复模块、背景基调、底部 action 区这种会影响整体观感与稳定性的点收束好，再考虑更细的动效；同时要补齐可逆操作（清空选择）来降低误操作成本。 |
| 2026-04-21 | 2.0-13 | “闭环按钮”比“更多页面”更能提升可演示性：在详情页加入“我做了”，让最近做过不再永久空；同时通过最小数据结构（去重+置顶）实现跨页联动，避免引入重状态库。 |
| 2026-04-21 | 2.0-14 | 响应式“看起来怪”往往不是组件坏了，而是布局触发点选错：桌面壳带侧栏时，内容区在 900~1100px 会非常窄，强行双栏会把步骤压成细长条；把断点提到更合理的宽度能显著提升稳定感。 |
| 2026-04-21 | 2.0-17 | Final QA 的重点不是“再加功能”，而是把可复现性与交付材料补齐：构建/启动/演示路径/检查清单/发布说明都要明确；同时要保证文档与常量一致（例如音效默认文件名），否则同学照做会失败。 |
